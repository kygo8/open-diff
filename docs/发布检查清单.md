# Release Checklist

This checklist is the required local gate before creating a release tag or publishing desktop
artifacts. Record command output, generated asset names, and reviewer notes in the release issue or
GitHub Release draft.

## 1. Version And Scope

- [ ] Confirm the release version is final and uses a semantic version tag such as `v1.0.0`.
- [ ] Confirm the version matches `package.json`, `src-tauri/Cargo.toml`, and
      `src-tauri/tauri.conf.json`.
- [ ] Confirm release scope, known limitations, and migration notes are captured in the release
      notes draft.

Evidence:

- Version diff or commit hash.
- Release notes draft link.
- Tag name to be created.

## 2. Quality Gate

- [ ] Run unit and browser smoke tests.

```bash
corepack pnpm test
```

- [ ] Run the full formatting, lint, type, Rust formatting, and Clippy gate.

```bash
corepack pnpm quality
```

- [ ] Run the full Rust workspace test suite.

```bash
cargo test --workspace --manifest-path src-tauri/Cargo.toml
```

Evidence:

- Command exit code `0`.
- Test counts from Vitest, Playwright, and Cargo.
- Any skipped tests with owner-approved reason.

## 3. Packaging Gate

- [ ] Build the default desktop bundle for the current platform.

```bash
corepack pnpm tauri:build
```

- [ ] Build the Windows portable package when preparing Windows assets.

```bash
corepack pnpm tauri:portable:windows
```

- [ ] Build Linux DEB assets when preparing Linux packages.

```bash
corepack pnpm tauri:build:linux:deb
```

- [ ] Build Linux RPM assets when preparing Linux packages.

```bash
corepack pnpm tauri:build:linux:rpm
```

- [ ] Build macOS app and DMG assets when preparing macOS packages.

```bash
corepack pnpm tauri:build:macos
```

Evidence:

- Artifact file names and sizes.
- Operating system and architecture used for each package.
- Checksums for uploaded assets.

## 4. Documentation Gate

- [ ] Confirm `README.md` describes the current feature set and install path.
- [ ] Confirm translated README files are in sync for supported languages:
      `docs/readme/README.zh-CN.md`, `docs/readme/README.zh-TW.md`,
      `docs/readme/README.ko.md`, `docs/readme/README.es.md`, `docs/readme/README.fr.md`, and
      `docs/readme/README.de.md`.
- [ ] Confirm `docs/需求文档.md` and `docs/开发手册.md` do not contradict the release scope.
- [ ] Confirm release notes mention new features, fixes, known issues, and upgrade guidance.

Evidence:

- Documentation review commit hash.
- Release notes preview.
- Any intentionally deferred translation updates.

## 5. License And Security Gate

- [ ] Confirm `LICENSE` is present in the repository and packaged assets.
- [ ] Confirm package metadata uses the expected `Apache-2.0` license value.
- [ ] Confirm `SECURITY.md` exists and vulnerability reporting instructions are current.
- [ ] Confirm bundled third-party notices or dependency reports are attached when required by the
      release owner.

Evidence:

- License file path in produced packages.
- Security contact or reporting path.
- Dependency/license report location if generated.

## 6. Release Execution

- [ ] Create or update the GitHub Release draft.
- [ ] Attach all generated artifacts and checksums.
- [ ] Confirm the release workflow in `.github/workflows/release.yml` completed successfully.
- [ ] Smoke test one downloaded artifact before publishing the release.

Evidence:

- GitHub Actions run URL.
- Release draft URL.
- Smoke-tested artifact name.

## 7. Rollback

- [ ] Keep the previous stable release URL available.
- [ ] Document the rollback trigger, owner, and communication channel.
- [ ] If a release is pulled, mark the GitHub Release as draft or prerelease and publish a short
      advisory with the replacement plan.

Evidence:

- Previous stable version.
- Rollback owner.
- User-facing notice location.
