# Accessibility Checklist

This checklist is the required accessibility gate for new views, major interaction changes, and
release review. Record the reviewer, tested operating system, browser or desktop build, and
evidence for every completed gate.

Scope:

- Text, folder, merge, table, hex, image, media, registry, remote, settings, and session views.
- Menus, toolbars, command palette, dialogs, status bar, progress panels, and report previews.
- Light, dark, and high-contrast themes.

## 1. Keyboard Access Gate

- [ ] Every primary workflow can be completed without a mouse.
- [ ] Tab order follows the visual reading order and never traps focus.
- [ ] Shift+Tab moves backward through the same reachable controls.
- [ ] Enter activates buttons, menu items, command palette results, and selected tree/table rows.
- [ ] Space toggles checkboxes, switches, segmented options, and selected row actions.
- [ ] Escape closes dialogs, popovers, command palette, context menus, and transient panels.
- [ ] Arrow keys navigate menus, tabs, tree rows, diff navigation, and grid-like controls where
      applicable.
- [ ] Focus returns to the invoking control after a dialog or popover closes.
- [ ] Disabled controls are skipped by Tab and expose a visible disabled state.

Evidence:

- Tested keyboard path for each changed workflow.
- Screenshot or short recording showing focus order for complex screens.
- List of intentional exceptions with owner approval.

## 2. Focus And Screen Reader Gate

- [ ] Icon-only buttons have an `aria-label` or an equivalent accessible name.
- [ ] Inputs, selects, search fields, sliders, and text areas have labels or labelled-by
      relationships.
- [ ] Dialogs use `role="dialog"` or platform equivalent, `aria-modal`, and a descriptive title.
- [ ] Progress indicators expose current value, total value, and status text when known.
- [ ] Status updates that require user awareness are announced through an accessible live region or
      a visible status message.
- [ ] Tables and tree views expose row, column, group, and selected/expanded state.
- [ ] Error messages identify the failed field or operation and include a recovery suggestion.

Evidence:

- Component or view test covering labels for newly added controls.
- Manual screen-reader smoke test for new dialogs, panels, or multi-step flows.

## 3. Non-Color Status Gate

- [ ] Difference states are never communicated by color alone.
- [ ] Added, removed, modified, unchanged, conflict, warning, error, and suppressed states include
      text, icon, shape, pattern, or explicit status labels.
- [ ] Folder sync actions such as copy, delete, overwrite, leave, and conflict include readable
      action text.
- [ ] Image, table, hex, and text comparison views expose status summaries outside the color layer.
- [ ] Legends and tooltips describe any color-coded status marks.

Evidence:

- Screenshot of changed status UI in light and dark themes.
- Test assertion or reviewer note proving each changed state has a non-color indicator.

## 4. High Contrast Gate

- [ ] The high-contrast theme is usable for navigation, editing, comparison, merge, and settings
      workflows.
- [ ] Text, icons, borders, focus rings, and disabled states meet WCAG 2.2 AA contrast expectations.
- [ ] Focus indicators remain visible against the focused component and surrounding background.
- [ ] Difference colors remain distinguishable when Windows high contrast or browser forced-colors
      mode is enabled.
- [ ] Custom color settings cannot produce unreadable text without showing a warning or reset path.

Evidence:

- Contrast measurement for new foreground/background pairs.
- Screenshot of changed UI in high-contrast mode.

## 5. Motion And Zoom Gate

- [ ] Essential information remains visible at 200% browser zoom or equivalent desktop scaling.
- [ ] Text can grow without overlapping toolbars, grids, panels, or buttons.
- [ ] Animations are non-essential or respect reduced-motion settings.
- [ ] Virtualized lists, diff maps, and large file views keep keyboard focus stable while scrolling.

Evidence:

- Screenshot at 200% zoom for changed dense views.
- Reviewer note for reduced-motion behavior when animation is introduced.

## 6. Review Outcome

Use this result block in pull requests and release notes:

```markdown
Accessibility review:

- Keyboard Access Gate: pass/fail/not applicable
- Focus And Screen Reader Gate: pass/fail/not applicable
- Non-Color Status Gate: pass/fail/not applicable
- High Contrast Gate: pass/fail/not applicable
- Motion And Zoom Gate: pass/fail/not applicable
- Evidence:
- Follow-up issues:
```

Failures:

- A failed gate blocks release unless the product owner accepts a documented exception.
- Exceptions must include impact, affected users, workaround, owner, and target fix version.
- Follow-up issues must reference the failing checklist section.
