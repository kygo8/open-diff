# Security Operation Audit Checklist

This checklist is the required security audit gate for any workflow that can modify local files,
remote resources, credentials, scripts, external commands, or release artifacts. Record the reviewer,
changed surface, threat assumptions, and Evidence for every gate.

Scope:

- Folder compare operations: copy, move, rename, delete, attribute changes, and touch.
- Folder sync operations: copy left/right, delete left/right, overwrite, and mirror sync.
- Remote profiles and remote credentials for FTP, SFTP, FTPS, WebDAV, S3, Dropbox, OneDrive, and
  Subversion.
- Script, command-line, silent script, external converter, and open-with workflows.
- Settings import/export and profile import/export.

## 1. Destructive Action Gate

- [ ] Every delete, overwrite, rename, move, mirror sync, and bulk attribute change has an explicit
      preview before execution.
- [ ] Destructive confirmation shows the side, source path, target path, item count, action type,
      and whether hidden or filtered items are included.
- [ ] delete and overwrite actions require a second confirmation when the action is irreversible or
      affects multiple items.
- [ ] mirror sync requires a summary of files to copy, files to overwrite, files to delete, conflicts,
      and errors.
- [ ] The user can cancel before execution and during long-running jobs.
- [ ] Safe write paths use temporary files and backups when supported by the target provider.
- [ ] Partial failure leaves original data intact or records which paths need manual recovery.
- [ ] No UI shortcut, command palette action, context menu action, CLI path, or script path bypasses
      the same confirmation policy.

Evidence:

- Screenshot or test output for the confirmation dialog.
- Operation log sample showing action, path, result, and error.
- Recovery note for partial failure handling.

## 2. Credential Handling Gate

- [ ] remote credentials are stored through a system keychain or approved credential-store
      abstraction.
- [ ] Profiles never serialize plaintext passwords, tokens, private keys, or refresh tokens by
      default.
- [ ] Profile export requires an explicit option before including secret material.
- [ ] Secret fields are masked in UI, logs, errors, reports, and screenshots.
- [ ] Clipboard copy of secrets is explicit, time-bound when possible, and never automatic.
- [ ] Test connection errors do not echo passwords, bearer tokens, access keys, or private-key
      fragments.
- [ ] Credential deletion explains which profiles or sessions will lose access.

Evidence:

- Serialized profile fixture proving secrets are represented by references.
- Log sample proving secrets are redacted.
- Manual review note for import/export behavior.

## 3. Script And Automation Gate

- [ ] A silent script cannot bypass file-system permissions, admin policy, confirmation policy, or
      provider-level authorization.
- [ ] Script commands that delete, overwrite, copy to protected paths, run sync, or write reports are
      visible in structured logs.
- [ ] Script variables are expanded deterministically and unknown variables fail closed.
- [ ] CLI and automation exit codes distinguish success, difference found, usage error, permission
      error, and execution error.
- [ ] Script execution records the script path, command line, line number, action, and result.
- [ ] Automation never persists credentials supplied on a command line unless the user explicitly
      saves a profile.

Evidence:

- Script log sample with line numbers and structured status.
- Test output for rejected unknown variables or forbidden operations.
- Exit-code contract reference.

## 4. External Command Gate

- [ ] External converters, open-with entries, and external applications show their configured source
      before first execution.
- [ ] Arguments are passed without shell interpolation unless the user intentionally configures a
      shell command.
- [ ] Input and output paths are quoted or passed as structured process arguments.
- [ ] External commands run with the least required working directory and environment variables.
- [ ] Failures are isolated: stderr, exit code, timeout, and produced output are captured without
      corrupting source files.
- [ ] Cached conversion output is invalidated when the source file, converter definition, or relevant
      settings change.
- [ ] Users can disable or remove external converter definitions.

Evidence:

- Converter definition review.
- Failure log containing exit code, stderr summary, timeout status, and output path.
- Confirmation that source files are not overwritten on failed conversion.

## 5. Remote Operation Gate

- [ ] Remote delete, overwrite, rename, and upload operations follow the same destructive action
      policy as local operations.
- [ ] Resume offsets and chunked transfer plans are validated before execution.
- [ ] Remote provider errors distinguish authentication, authorization, missing path, conflict,
      network, and backend failures.
- [ ] Network retries do not duplicate destructive operations.
- [ ] Provider-specific path normalization cannot escape the configured remote root.

Evidence:

- Transfer plan or log sample with operation IDs.
- Provider error sample with redacted endpoint and credential reference.
- Test output for invalid resume offsets or protected overwrite policy.

## 6. Audit Trail Gate

- [ ] Security-sensitive actions emit structured logs with timestamp, domain, action, path, status,
      and error details.
- [ ] Logs never contain plaintext secrets.
- [ ] Reports generated from logs preserve enough context for support without exposing private
      content unnecessarily.
- [ ] User-facing errors include recovery guidance and a stable error code.

Evidence:

- Redacted structured log sample.
- Error payload sample with stable code and localized message key.

## 7. Review Outcome

Use this result block in pull requests and release notes:

```markdown
Security operation audit:

- Destructive Action Gate: pass/fail/not applicable
- Credential Handling Gate: pass/fail/not applicable
- Script And Automation Gate: pass/fail/not applicable
- External Command Gate: pass/fail/not applicable
- Remote Operation Gate: pass/fail/not applicable
- Audit Trail Gate: pass/fail/not applicable
- Evidence:
- Follow-up issues:
```

Failures:

- A failed destructive-action, credential, silent script, or external command gate blocks release.
- Exceptions require impact, affected workflow, compensating control, owner, and target fix version.
- Follow-up issues must reference the failing checklist section and include a reproduction path.
