## 1. 执行规则

### 1.1 子任务定义

每个子任务必须满足：

- 有明确输入、输出和验收标准。
- 可在 1 到 4 小时内完成；超过 4 小时必须继续拆分。
- 可独立验证；无法独立验证的任务必须拆到可验证边界。
- 完成后必须更新本文档中的“进度”。
- 完成后必须单独提交一次代码，不能多个子任务合并成一个提交。

### 1.2 进度状态

| 状态               | 含义                                    |
| ------------------ | --------------------------------------- |
| 未开始             | 尚未编码或设计                          |
| 进行中             | 当前正在开发                            |
| 已完成             | 代码、测试、文档、提交均已完成          |
| 已完成（待补提交） | 功能已在当前工作区存在，但缺少 Git 提交 |
| 阻塞               | 依赖外部决策、权限、技术验证或资源      |
| 延后               | 明确不进入当前阶段                      |

### 1.3 每个子任务的标准流程

1. 将任务状态改为“进行中”。
2. 完成最小设计或测试用例。
3. 编码实现。
4. 运行该任务相关的最小验证命令。
5. 运行通用质量门禁。
6. 将任务状态改为“已完成”。
7. 提交代码，提交信息必须使用中文。

推荐提交命令：

```powershell
git add .
git commit -m "完成：<任务编号> <任务名称>"
```

### 1.4 通用验证命令

每个任务至少运行相关子集。涉及前后端集成、共享类型、Tauri command、构建脚本时必须运行完整门禁。

```powershell
corepack pnpm quality
corepack pnpm test
corepack pnpm build
Set-Location src-tauri
cargo test --workspace
```

涉及发行打包或 Tauri 配置时额外运行：

```powershell
corepack pnpm tauri:build
```

## 18. 子任务完成检查单

每个子任务提交前必须逐项确认：

- [ ] 本文档中对应任务状态已从“进行中”改为“已完成”。
- [ ] 代码只覆盖当前子任务范围。
- [ ] 前端代码通过格式化、ESLint、Stylelint、类型检查。
- [ ] Rust 代码通过 fmt、clippy、相关单元测试。
- [ ] 新增能力有最小测试或可复现手工验证步骤。
- [ ] 涉及危险操作时有二次确认、取消、错误展示。
- [ ] 提交信息为中文，格式为 `完成：<任务编号> <任务名称>`。
