# Performance Budget

This document defines the measurable performance budget for comparison workflows. Any change that
touches loading, diffing, folder scanning, virtual scrolling, sync planning, report rendering, or
remote transfer planning must record Evidence against the relevant budget.

Reference fixture:

- `performance-core` text benchmark: at least 20,000 lines per side.
- `performance-core` folder benchmark: at least 5,000 entries per side.
- `performance-core` binary benchmark: 2 MiB payload read in 64 KiB chunks.

Environment baseline:

- Local development machine or CI runner used for the pull request.
- Release builds are preferred for timing-sensitive checks.
- If debug builds are used, record that in Evidence and compare only against previous debug-build
  results.

## 1. Large File Budget

- [ ] Text compare starts progressive rendering within 500 ms for 20,000 lines per side.
- [ ] Text compare produces first visible diff rows within 1,500 ms for 20,000 lines per side.
- [ ] Binary compare reads large files in chunks and does not require loading files larger than the
      active block window into UI state.
- [ ] The 2 MiB binary benchmark completes comparison without UI-thread blocking longer than 100 ms.
- [ ] Memory growth is proportional to the active comparison model and does not duplicate full file
      content in both UI and worker state unless explicitly justified.
- [ ] Loading status stays visible for operations expected to exceed 300 ms.

Evidence:

- Benchmark command or manual profiling notes.
- Time to first visible rows.
- Peak memory estimate or profiler screenshot for changed large-file code.

## 2. Folder Scan Budget

- [ ] Folder scanning runs in the background and can be cancelled.
- [ ] The first 5,000 local entries are listed or streamed into the model within 2,000 ms on the
      baseline machine.
- [ ] Content scan updates status incrementally instead of waiting for the full tree to finish.
- [ ] Folder scan cancellation is observed within 250 ms after the cancel request reaches the worker.
- [ ] Remote scans cache metadata and expose an explicit refresh path.
- [ ] Scan errors preserve partial results and identify the failed path.

Evidence:

- Scan timing for the 5,000-entry benchmark or an equivalent fixture.
- Cancellation timing note.
- Log sample for partial success and failed path reporting.

## 3. Scrolling Budget

- [ ] Text, table, hex, and folder views maintain 60 FPS during ordinary vertical scrolling on the
      baseline machine.
- [ ] Synchronized panes remain within one row or one rendered block of each other after wheel,
      keyboard, scrollbar, and programmatic navigation.
- [ ] A single scroll frame should complete within 16 ms; frames above 50 ms require investigation.
- [ ] Virtualized lists preserve focused row, selected row, and diff navigation target while new rows
      mount or unmount.
- [ ] Horizontal scrolling in hex and table views does not trigger full diff recomputation.
- [ ] Difference map and minimap updates are throttled or batched for large documents.

Evidence:

- Browser performance trace, desktop profiler capture, or manual FPS note.
- View name, row count, selected filters, and theme.
- Any accepted dropped-frame reason with follow-up issue.

## 4. Report Generation Budget

- [ ] Text, folder, table, hex, registry, media, and version reports render without blocking the UI
      thread for more than 100 ms at a time.
- [ ] Reports for 20,000-line text comparisons begin streaming or show progress within 1,000 ms.
- [ ] Reports for 5,000-entry folder comparisons complete or produce a downloadable partial artifact
      within 5,000 ms on the baseline machine.
- [ ] HTML report generation escapes content while preserving file links and does not embed
      unbounded raw binary content.
- [ ] JSON and XML report generation stays deterministic for the same comparison result.
- [ ] Report cancellation or close behavior is defined for long-running jobs.

Evidence:

- Report type, fixture size, output size, and elapsed time.
- Generated artifact path or checksum.
- Log entry for report start, success, failure, or cancellation.

## 5. Regression Policy

- [ ] A pull request fails the performance gate when a budget regresses by more than 20% without an
      approved reason.
- [ ] Intentional regressions must include user impact, mitigation, owner, and target recovery
      version.
- [ ] New views must add a budget row before release if they introduce loading, scanning, scrolling,
      or report generation behavior.
- [ ] Performance-sensitive code paths should prefer streaming, chunking, virtualized rendering,
      caching, cancellation, and structured progress events.

## 6. Review Outcome

Use this result block in pull requests and release notes:

```markdown
Performance budget review:

- Large File Budget: pass/fail/not applicable
- Folder Scan Budget: pass/fail/not applicable
- Scrolling Budget: pass/fail/not applicable
- Report Generation Budget: pass/fail/not applicable
- Evidence:
- Follow-up issues:
```

Failures:

- A failed large-file, scan, scrolling, or report generation budget blocks release unless the product
  owner accepts a documented exception.
- Follow-up issues must include the measured baseline, regression value, affected workflow, and
  expected recovery target.
